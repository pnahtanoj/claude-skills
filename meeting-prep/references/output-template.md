# Output Template Detection

The skill matches the team's existing template — it does not impose its own. This file documents how to detect, common variants, and how to handle teams without a standardized agenda.

## Why detection matters

A team that runs an EOS-style L10 has specific section names: **Segue, Scorecard, Rock Review, Customer/Employee Headlines, To-Do List, IDS** (Identify, Discuss, Solve). A team that runs a "weekly leadership sync" might use different names: **Last week, This week, Blockers, Decisions needed**. Forcing the EOS shape onto a team that doesn't use EOS makes the draft feel foreign and signals to teammates that the user isn't really in the meeting — they're pasting AI output.

## Detection procedure

In Phase 1, after reading the agenda doc:

1. **Identify the user's sections** by looking for headings that match the user's name (in `agenda_doc.sections` or by scanning for the user's name as a heading or sub-heading).
2. **Read the previous week's content** in those sections to learn the team's vocabulary. The headings the team actually uses are more authoritative than what's in `config.agenda_doc.sections` — config can drift.
3. **Capture the section order** as it appears in the doc. The order is part of the template.
4. **Note any team-specific conventions**:
   - Are to-dos prefixed with names? (e.g., *"[JP] Investigate billing deadlock"*)
   - Are dates inline or in a separate column?
   - Are statuses in brackets, parentheses, or as their own column?
   - Are bullet glyphs `-`, `*`, or `•`?

## Common variants

### EOS L10 (most common in mid-stage companies)

```markdown
## Segue
[Each person: one personal best, one professional best from the week]

## Scorecard review
[Numbers]

## Rock review
[Each rock: on-track / off-track + one-line update]

## Customer/Employee Headlines
[Breakthroughs, fires, news]

## To-Do List
[Last week's to-dos with status, then new to-dos with owner + date]

## IDS (Identify, Discuss, Solve)
[Issues to discuss, with structured framing]
```

The user's section in this format is typically a sub-heading under each top-level section — e.g., under To-Do List, "Jonathan's to-dos".

### Weekly leadership sync (custom)

```markdown
## [Date] — Weekly Leadership Sync

### Jonathan
- Last week:
- This week:
- Blockers / Decisions needed:
- Headlines:
```

Less structured but still per-person sections.

### Hybrid / minimal

Some teams have very thin agendas — just a running notes doc with names as sub-headings. In this case, treat the user's last-week sub-section as the implicit "last week's to-dos" and look for any bulleted commitments to status against.

### No template

If the doc has no per-person sections and no recurring structure, the skill cannot work. Flag to the user: *"I can't detect a template in the agenda doc — there's no structured section for last week's commitments to anchor against. Want to set one up, or is this not actually a recurring-agenda meeting?"*

## What to preserve from the existing template

- **Section names** — verbatim, including capitalization.
- **Section order** — match.
- **Sub-section conventions** — if the team uses `### Jonathan` for the user's section, do the same.
- **Bullet style** — match the prevailing glyph in the team's portion of the doc.
- **Status notation** — if statuses are in `[brackets]`, use brackets. If they're after a `→`, use that.
- **Date format** — match. If the team writes `4/23`, don't write `2026-04-23`. If they write `2026-04-23`, don't write `4/23`.

## What NOT to add

- **Don't add sections the team doesn't use.** If they don't have an IDS section, don't create one — surface IDS-worthy items as a Headline or a Decision needed using the team's own framing.
- **Don't add framing the team doesn't use.** Don't introduce "Decision needed / Options / Open Qs" structure if the team has been listing decisions as plain bullets and that's working for them. (Exception: if the user explicitly asks for IDS framing, use it.)
- **Don't add metadata.** No "draft generated by…", no timestamps, no "for review" headers. The output is paste-ready.

## Sub-section: the user's section heading

If the user's section in the doc is typically `### Jonathan's update` or similar, the draft's top-level section is that heading. Don't wrap the draft in an outer "Meeting Prep Draft" heading — that just gets stripped when the user pastes.

The draft chat output can have a one-line preamble (*"Draft for L10 on 2026-04-25, anchored in 8 commits / 2 docs / 3 Granola transcripts."*) but the markdown body should be paste-ready.

## When the team's template conflicts with anti-slop rules

The team's template wins on structure. The anti-slop rules win on content. So:
- If the team uses three-bullet patterns ("Last week / This week / Blockers"), keep the three-bullet structure even though anti-slop rule 4 warns against triples — the template is authentically three-fold, not a model bias.
- If the team's archived prior weeks are full of slop ("comprehensive alignment", "leveraged…"), the user's draft should NOT match that style. Style decisions follow the anti-slop rules; structure follows the template.
